## 1. Model-Centric AI의 정의

대부분의 머신러닝 교육은 하나의 가정 하에 이루어진다. 데이터는 이미 잘 정제되어 있고, 우리가 해야 할 일은 그 데이터에 가장 잘 맞는 모델을 찾는 것이라는 가정이다.

강의에서 제공되는 데이터셋은 대체로 오류가 없고, 라벨이 정확하며, 분포 역시 안정적이다. 수업 자료로 유명한 고양이/강아지 이미지 분류와 같은 전형적인 예제에서는 데이터 자체가 문제의 원인이 되는 경우가 거의 없다. 따라서 학습자의 관심은 자연스럽게 모델로 향한다.

이 환경에서 머신러닝의 핵심 문제는 이제 주어진 데이터셋에 대해 어떤 모델 구조와 학습 전략을 선택해야 성능을 최대화할 수 있는가이다. 이 문제에 답하기 위해 우리는 신경망 구조, 결정 트리, 정규화 기법 등 모델 자체를 개선하는 기술들을 배운다. MIT DCAI는 이러한 접근을 **Model-Centric AI**라고 정의한다.

---

## 2. 실무에서의 머신러닝이 이론과 괴리되는 이유

실무에서의 머신러닝 문제는 교육용 예제와 구조적으로 다르다. 실무 환경에서의 데이터는 고정된 입력이 아니다. 기업이나 서비스 환경에서는 데이터를 수정할 수 있고, 필요하다면 추가 데이터를 수집할 수도 있으며, 심지어 데이터 정의 자체를 다시 설계하는 것도 가능하다.

더 중요한 점은, 현실의 데이터는 거의 예외 없이 지저분하고, 불완전하며, 오류를 포함하고 있다는 사실이다. 그 예로 현실 데이터는 라벨이 틀려 있거나, 관측 값이 누락되어 있거나, 데이터 분포가 시간에 따라 변하거나, 수집 과정에서 체계적인 편향이 발생할 수 있다.

이러한 상황에서는 아무리 정교한 모델을 사용하더라도 성능 향상에 한계가 발생한다. 이에 대하여 경험이 풍부한 데이터 과학자들이 공통적으로 갖는 직관은 다음과 같다.

> 모델을 더 복잡하게 만드는 것보다, 데이터를 이해하고 고치는 것이 성능 개선에 훨씬 큰 영향을 준다.

하지만 문제는 데이터의 규모다. 데이터셋이 커질수록, 데이터를 하나하나 살펴보며 수정하는 방식은 더 이상 현실적인 선택지가 아니다. 그럼에도 불구하고 전통적으로 데이터 개선은 사람의 직관에 의존한 ad-hoc 수작업으로 이루어져 왔다.

## 3. Data-Centric AI의 문제 재정의

MIT DCAI가 제안하는 전환은 문제 정의 자체를 바꾸는 것이다. 모델을 중심에 두는 대신, 데이터를 시스템적으로 다루는 것을 연구의 중심으로 끌어온다. Data-Centric AI는 단순히 데이터를 잘 관리하는 것이 아니라, 데이터에 존재하는 구조적 문제를 알고리즘적으로 탐지하고 수정하고자 한다.

탐색적 데이터 분석(EDA)은 여전히 중요한 첫 단계이다. 그러나 EDA는 본질적으로 사람이 직접 관찰하고 해석하는 과정으로, 대규모 데이터 환경에서는 확장성이 떨어진다.

Data-Centric AI는 AI 기법 자체를 이용해 데이터 문제를 진단하고 개선하려는 시도이다. 이 접근은 크게 두 가지 방향으로 나타난다.

- **첫째**, 데이터의 난이도나 특성을 이해하여 모델 학습 과정을 조정하는 방식이다. 대표적인 예로 Curriculum Learning으로, 모델이 쉬운 데이터부터 점진적으로 학습하도록 유도한다.
- **둘째**, 데이터 자체를 수정하여 모델 성능을 개선하는 방식이다. Confident Learning은 모델의 출력 정보를 활용해 잘못 라벨링된 데이터를 찾아 제거하거나 수정한다.

이 두 접근의 공통점은 명확하다.  
**어떤 데이터가 문제인가에 대한 판단을 사람이 아니라 알고리즘이 수행한다는 점이다.**

## 4. Data-Centric 관점의 파이프라인

Model-Centric AI와 Data-Centric AI의 차이는 목표 정의에서 가장 분명하게 드러난다.

| 구분 | 목표 |
|---|---|
| Model-Centric AI | 주어진 데이터셋에 대해 최고의 모델을 만든다 |
| Data-Centric AI | 주어진 모델이 학습하기에 가장 좋은 데이터셋을 만든다 |

실전 시스템에서는 이 둘을 분리할 수 없다. MIT DCAI는 이를 하나의 반복적인 파이프라인으로 제시한다.

1. 데이터를 탐색하고, 명백한 오류와 구조적 문제를 수정한다.  
2. 정제된 데이터로 baseline 모델을 학습한다.  
3. 이 모델을 사용해 데이터의 문제를 다시 진단한다.  
4. 개선된 데이터로 모델을 다시 학습한다.

중요한 점은, 성능 개선의 핵심 반복 루프가 **모델 튜닝이 아니라 데이터와 모델의 상호작용**이라는 점이다. 단순히 모델만 바꾸는 접근은 장기적으로 성능 한계에 부딪힌다.

## 5. Data-Centric AI를 이용한 문제 해결 방법

Data-Centric AI는 특정 알고리즘 하나를 가리키는 용어가 아니라, 문제를 바라보는 관점의 집합에 가깝다. 이 관점 아래에는 다양한 방법들이 포함된다.

- 이상치 탐지: 분포에서 벗어난 데이터를 식별  
- 오류 탐지: 잘못된 값이나 라벨을 자동으로 탐지  
- 합의 추정: 여러 주석자 간의 불일치 해결  
- 데이터 증강: 사전 지식을 데이터 형태로 주입  
- feature 엔지니어링: 표현 공간 자체를 재설계  
- 능동 학습: 가장 정보량이 큰 데이터를 선택  
- 커리큘럼 학습: 학습 순서를 구조화  

이 기법들의 공통 목표는 하나다.  
**모델이 더 좋은 데이터를 활용할 수 있도록 만드는 것이다.**

## 6. Data-Centric AI가 각광받는 이유

잘못된 라벨이 포함된 데이터는 모델이 아무리 커져도 그대로 학습된다.  
이 문제는 모델 규모의 문제가 아니라 데이터 품질의 문제다.

저자들은 Data-Centric AI의 필요성을 단순한 성능 문제가 아니라 사회적·경제적 문제로 확장한다.

잘못된 데이터는 미국에서만 매년 약 3조 달러의 비용을 유발한다. 데이터 품질 문제는 거의 모든 산업에 존재하며, 데이터 규모가 커질수록 수작업 기반의 해결은 불가능해진다.

ChatGPT와 같은 대규모 모델조차 낮은 품질의 데이터에서 발생하는 문제를 완화하기 위해 막대한 Human Review에 의존했다. 하지만 저자들은 이러한 방식이 근본적인 해결책이 아니라고 지적한다.

머신러닝이 의료, 금융, 교통과 같이 실패 비용이 큰 영역에 깊이 통합되는 상황에서, 모델이 어떤 데이터로 학습되었는지 자체를 보장하는 체계가 필요하다.

---

## 7. 실증적 근거: Noisy Labels 실험

Data-Centric AI의 효과는 실험적으로도 확인된다. CIFAR-10 데이터셋에서 라벨 노이즈를 점점 증가시키며 수행한 벤치마크 연구에서는, 노이즈를 모델 내부에서 보정하려는 복잡한 방법보다 데이터셋을 적응적으로 수정하는 단순한 방법이 더 높은 정확도를 달성했다.

이 결과는 모델의 복잡성보다 데이터의 품질이 더 중요하다는 사실을 보여준다.

## 실무에서는 모델의 비중보다 데이터셋의 비중이 더 크다

현실의 머신러닝 개발 환경에서는 이상적인 데이터셋을 처음부터 새로 구축하는 것이 시간적·비용적으로 거의 불가능한 경우가 많다. 대부분의 실무 프로젝트는 이미 존재하는 대규모 데이터셋(웹에서 수집된 텍스트, 로그 데이터, 크라우드소싱 라벨, 공개 벤치마크)에 의존하며 출발한다.

이러한 데이터는 규모가 방대하지만 여러 구조적 문제를 내재하고 있으며, 이러한 결함을 사후적으로 완벽하게 정제하는 것이 현실적으로 불가능하다는 단점을 가지고 있다. 이러한 맥락에서 Data-Centric AI는 이미 존재하는 불완전한 대규모 데이터 환경을 전제로 한 현실적인 해법으로 이해되어야 한다.

더 나아가, 최근의 Data-Centric AI는 단순히 데이터를 정제하는 데 그치지 않고, 모델과 상호작용하며 새로운 학습 신호를 생성하는 방식으로 데이터를 확장하고 재구성하는 단계로 나아가고 있다. 즉, 더 나은 머신러닝 시스템을 위해 필요한 것은 데이터를 새롭게 생성, 재해석, 재조합할 수 있는 체계적 방법이다.

결국, 거대 모델과 거대 데이터셋이 이미 표준이 된 현재의 머신러닝 환경에서 앞으로의 발전을 좌우하는 핵심 요소는 데이터를 어떻게 만들어내고 활용할 것인가에 대한 Data-Centric AI의 진화라고 할 수 있다.

**Data-Centric AI는 성능 개선 기법이 아니라, 머신러닝 시스템을 바라보는 기본 좌표계를 바꾸는 시도이다.**
